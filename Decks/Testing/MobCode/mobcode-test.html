Activity Testing

MobCode

Test deck for the mobcode embedded activity: four file-seeding variants across five slides.

Background

How MobCode Works

The mobcode activity opens a shared live-coding workspace. The deck can seed it with starter files via the files payload field. Once a session exists, the live state is authoritative and re-entering the slide does not overwrite instructor edits.

Key payload fields

  • files — object mapping virtual path to UTF-8 content. Omit entirely for an empty workspace.
  • activeFile — optional path that should be focused when the workspace opens. Must match a key in files.
  • Path traversal segments such as ../ are rejected and will not load.

What this deck tests

  • Test A: empty workspace (no files key).
  • Test B: single Java file with activeFile.
  • Test C: two Java files with activeFile pointing to the second file.
  • Test D: three-file classroom exercise (Java + README) with activeFile on Main.java.
Test A

Empty Workspace

No files field is provided in the payload. The activity should open a blank MobCode workspace with no pre-seeded files.

Expected behavior

  • The MobCode workspace opens with an empty file tree.
  • No files are shown in the editor; participants start from scratch.
  • Files created during the session persist in the live state and are not overwritten when the slide is re-entered.

Payload

JSON
{
  // no "files" key — empty workspace
}
Test B

Single File + activeFile

One file is seeded (src/Main.java) and activeFile is set to the same path. This is the simplest real starter scenario.

Expected behavior

  • The workspace opens with src/Main.java already visible in the editor.
  • The file contains a minimal Hello World program that compiles and prints Hello, MobCode!
  • Re-entering the slide after session creation must not reset the file contents to the original starter.

Payload

JSON
{
  "files": {
    "src/Main.java":
      "public class Main {\n
  public static void main(
      String[] args) {\n
    System.out.println(\n
      \"Hello, MobCode!\");\n
  }\n}\n"
  },
  "activeFile": "src/Main.java"
}
Test C

Two Files, activeFile on Second

Two Java files are seeded. activeFile is set to src/Greeter.java, the second file in the map, so the editor should open on that file rather than on Main.java.

Expected behavior

  • Both src/Main.java and src/Greeter.java appear in the file tree.
  • The editor opens with src/Greeter.java active (not Main.java), verifying that activeFile is respected even when it is not the first key.
  • Running the full program should print Hello, World!

Payload

JSON
{
  "files": {
    "src/Main.java": "...",
    "src/Greeter.java":
      "public class Greeter {\n
  public String greet(
      String name) {\n
    return \"Hello, \"\n
      + name + \"!\";\n
  }\n}\n"
  },
  // activeFile points to the 2nd file
  "activeFile": "src/Greeter.java"
}
Test D

Three-File Classroom Exercise

A realistic starter kit: two stub Java classes and a README with instructions. Students complete Counter.increment() and wire it up in Main. activeFile opens the editor on Main.java.

Expected behavior

  • Three files appear in the file tree: src/Main.java, src/Counter.java, and README.md.
  • The editor opens on src/Main.java showing the // TODO stub.
  • Once complete, the program should compile and print 3.
  • The README renders as a markdown file and is readable from the file tree.

Files (abbreviated)

JSON
{
  "files": {
    "src/Main.java":
      // stub — TODO to create Counter
      "public class Main { ... }",
    "src/Counter.java":
      // stub — TODO increment()
      "public class Counter { ... }",
    "README.md":
      "# Counter Exercise\n..."
  },
  "activeFile": "src/Main.java"
}
