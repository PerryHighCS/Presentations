Activity Testing

MobCode

Test deck for the mobcode embedded activity: four file-seeding variants plus two special-case Python tests across six slides.

Background

How MobCode Works

The mobcode activity opens a shared live-coding workspace. The deck can seed it with starter files via the files payload field. Once a session exists, the live state is authoritative and re-entering the slide does not overwrite instructor edits.

Key payload fields

  • files — object mapping virtual path to UTF-8 content. Omit entirely for an empty workspace.
  • activeFile — optional path that should be focused when the workspace opens. Must match a key in files.
  • Path traversal segments such as ../ are rejected and will not load.

What this deck tests

  • Test A: empty workspace (no files key).
  • Test B: single Python file with activeFile.
  • Test C: two Python files with activeFile pointing to the second file.
  • Test D: three-file classroom exercise (Python + README) with activeFile on main.py.
  • Test E: input() special case — blocking stdin prompt.
  • Test F: time.sleep() special case — blocking delay.
Test A

Empty Workspace

No files field is provided in the payload. The activity should open a blank MobCode workspace with no pre-seeded files.

Expected behavior

  • The MobCode workspace opens with an empty file tree.
  • No files are shown in the editor; participants start from scratch.
  • Files created during the session persist in the live state and are not overwritten when the slide is re-entered.

Payload

JSON
{
  // no "files" key — empty workspace
}
Test B

Single File + activeFile

One file is seeded (main.py) and activeFile is set to the same path. This is the simplest real starter scenario.

Expected behavior

  • The workspace opens with main.py already visible in the editor.
  • The file contains a minimal Hello World program that runs and prints Hello, MobCode!
  • Re-entering the slide after session creation must not reset the file contents to the original starter.

Payload

JSON
{
  "files": {
    "main.py":
      "print(\"Hello, MobCode!\")\n"
  },
  "activeFile": "main.py"
}
Test C

Two Files, activeFile on Second

Two Python files are seeded. activeFile is set to greeter.py, the second file in the map, so the editor should open on that file rather than on main.py.

Expected behavior

  • Both main.py and greeter.py appear in the file tree.
  • The editor opens with greeter.py active (not main.py), verifying that activeFile is respected even when it is not the first key.
  • Running the full program should print Hello, World!

Payload

JSON
{
  "files": {
    "main.py": "...",
    "greeter.py":
      "class Greeter:\n
    def greet(self, name):\n
        return f\"Hello, {name}!\"\n"
  },
  // activeFile points to the 2nd file
  "activeFile": "greeter.py"
}
Test D

Three-File Classroom Exercise

A realistic starter kit: two stub Python files and a README with instructions. Students complete Counter.increment() and wire it up in main.py. activeFile opens the editor on main.py.

Expected behavior

  • Three files appear in the file tree: main.py, counter.py, and README.md.
  • The editor opens on main.py showing the # TODO stub.
  • Once complete, the program should run and print 3.
  • The README renders as a markdown file and is readable from the file tree.

Files (abbreviated)

JSON
{
  "files": {
    "main.py":
      # stub — TODO to create Counter
      "from counter import Counter\n...",
    "counter.py":
      # stub — TODO increment()
      "class Counter: ...",
    "README.md":
      "# Counter Exercise\n..."
  },
  "activeFile": "main.py"
}
Test E — Special Case

input() Blocking Prompt

The program calls input(), which blocks execution until the user types a response and presses Enter. This tests that the workspace correctly surfaces a live stdin prompt rather than hanging silently.

Expected behavior

  • Running the program displays the prompt Enter your name: in the console and waits for input.
  • After the user types a name and presses Enter, the program prints Hello, <name>! and exits normally.
  • The workspace must not freeze, time out, or skip the prompt silently.

Seeded file: main.py

Python
name = input("Enter your name: ")
print(f"Hello, {name}!")
Test F — Special Case

time.sleep() Blocking Delay

The program calls time.sleep(2), which suspends execution for two seconds before printing a second message. This tests that the runtime tolerates a deliberate blocking delay without killing the process or reporting a timeout error.

Expected behavior

  • The console immediately prints Starting... then pauses for approximately two seconds.
  • After the delay, the console prints Done after 2 seconds! and the process exits normally.
  • The workspace must not report an error or kill the process during the sleep interval.

Seeded file: main.py

Python
import time

print("Starting...")
time.sleep(2)
print("Done after 2 seconds!")
